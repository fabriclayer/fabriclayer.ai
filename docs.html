Fabric Documentation

The Fabric Trust Index is a public directory of trust scores for AI tools, models, MCP servers, and API services. Over 700 indexed services each receive a composite score from 0.00 to 5.00, computed from six independently weighted signals — all generated automatically from public data sources.

Trust Score

The Fabric Trust Score is the core of the Trust Index. It determines whether a service is trusted, flagged for caution, or blocked outright.

Every AI service indexed by Fabric receives a composite score from 0.00 to 5.00, computed from six independently weighted signals. Each signal scores 0–5 individually, then is multiplied by its weight. The six weighted contributions sum to the final score.

All scoring data is generated automatically from public sources. No manual reviews. No paid placements. No provider involvement required. A provider doesn't need to opt in — or even know they're being scored — for Fabric to evaluate them.

Why six signals? Any single metric can be gamed. Download counts can be inflated. GitHub stars can be bought. Even vulnerability databases have blind spots. By combining six independent signals across security, operations, maintenance, adoption, transparency, and publisher identity — an attacker would need to simultaneously spoof all of them to pass. That's the point.

The six signals

1. Vulnerability & Safety

The highest-weighted signal. Measures known security vulnerabilities across the package and its entire dependency tree using the OSV.dev CVE database.

Fabric scans for open CVEs at every severity level — critical, high, medium, and low. The score starts at 5.0 and deducts based on severity: an unpatched critical CVE deducts 2.0 points and floors the score at 1.0. Patched vulnerabilities still incur smaller deductions (e.g. −0.3 for a patched critical) because they indicate the dependency was once vulnerable.

Supply chain depth matters. Fabric traces the full dependency tree from npm and PyPI. Transitive vulnerabilities (CVEs in dependencies of dependencies) are counted at 25% of the direct rate, capped at −0.25 total. A package with zero direct vulnerabilities can still score poorly if its dependencies are compromised.

2. Operational Health

The only signal generated by Fabric's own infrastructure. We actively ping service endpoints every 15 minutes for uptime, latency, and HTTP status — no provider can fake this one.

Services with a real, publicly reachable endpoint can score up to 5.0. Services that only have a registry listing (e.g. an npm page with no live API) are capped at 4.5. Uptime is the primary driver: 99.9%+ uptime scores 5.0, 99.5%+ scores 4.5, 99%+ scores 4.0, and 95%+ scores 3.0.

Services with no endpoint to monitor (CLI tools, local packages) receive a fallback value rather than zero, which prevents them from being unfairly penalised.

3. Maintenance Activity

Is anyone home? Abandoned software is a security risk regardless of how clean the code is today. This signal captures the human factor behind a project.

Fabric evaluates four factors: commit recency (last commit within 7 days → 5.0, within 30 days → 4.0, over 365 days → 1.0), commit frequency (50+ commits in 90 days → 4.5), issue responsiveness (median response under 24 hours → +0.5), and release cadence (monthly releases → +0.3). These combine into the signal score.

Stable, mature projects with low commit frequency aren't penalised as heavily — release cadence and issue responsiveness carry more weight than raw commit volume.

4. Adoption

Not a security signal per se, but a strong proxy for how many eyes are on the project. High adoption means more users finding and reporting issues, more pressure to maintain quality, and a larger blast radius if something goes wrong — which motivates faster fixes.

Download counts are scored on logarithmic tiers: ≥10M weekly downloads scores 5.0, ≥1M scores 4.5, ≥100K scores 4.0, ≥10K scores 3.5, and ≥1K scores 3.0. Download velocity matters too — growth over 20% week-over-week adds +0.5, while decline over 20% deducts −0.5. For categories with 10+ services, scores are normalised so niche tools aren't penalised for being specialised.

Adoption is gameable. This is the one signal attackers can most easily inflate — which is why it's weighted lower than the safety and operational signals. The other five signals act as independent checks.

5. Transparency

Can you see what it does before you run it? Open source projects with clear documentation, proper licensing, and published schemas are inherently more trustworthy than black boxes.

Fabric evaluates a checklist of transparency items: public source code, OSI-approved licence, documentation with code examples, SECURITY.md file, and API documentation. For AI model categories (LLMs, image generation, speech, vision), a model card is also required as a sixth checklist item. Each item contributes equally to the signal score.

6. Publisher Trust

Who built this? A first-time anonymous account publishing a critical infrastructure tool is a very different risk profile than a verified organisation with many other maintained packages.

Fabric evaluates four factors: account age (≥5 years scores highest), organisation type (org accounts score higher than personal), publishing track record (number of maintained public repositories), and identity consistency (whether the publisher name matches across registries like GitHub and npm). The raw score is normalised to the 0–5 scale.

How scoring works

The final score is a weighted sum of all six signals. Each signal is scored independently on a 0–5 scale, then multiplied by its weight. The weights sum to 1.00, producing a final composite score between 0.00 and 5.00.

formula
composite = vuln×0.25 + maint×0.20 + ops×0.15 + adopt×0.15 + transp×0.15 + pub×0.10
SignalWeightPrimary source
Vulnerability & Safety0.25OSV.dev CVE database
Maintenance Activity0.20GitHub API
Operational Health0.15Fabric Monitor (HTTP pings)
Adoption0.15npm / PyPI download stats
Transparency0.15GitHub repo analysis
Publisher Trust0.10GitHub + npm identity

When a signal has no data available (e.g. no npm package for download stats), the system applies a fallback value rather than zero — this prevents unfair penalisation for services that simply don't have a presence on a particular registry.

How to improve a score. Fix known vulnerabilities. Maintain uptime. Keep the project active with regular releases. Improve documentation. Verify your publisher identity. There are no shortcuts — the only way to raise a score is to improve the underlying signals.

Score thresholds

Every indexed service falls into one of three tiers based on its composite score. These labels are shown publicly on the Trust Index.

ScoreStatusMeaning
3.25 – 5.00TrustedLow risk across all signals. Auto-approve for agent use.
1.00 – 3.24CautionSome signals flagged. Require human confirmation before granting access.
0.00 – 0.99BlockedSignificant risks detected. Deny by default.

Additional override rules may force a status downward (never upward). For example, any service with an unpatched critical CVE is automatically capped at Blocked regardless of its other signals. Services with no available data sources are held in a Pending state until initial collection completes.

Data freshness

The scoring engine runs on a fixed schedule across all indexed services. Different signals are refreshed at different intervals depending on how quickly the underlying data changes.

JobScheduleWhat it does
Health checksEvery 15 minutesHTTP pings to all service endpoints for uptime and latency
CVE scanningHourlyChecks OSV.dev for new vulnerabilities affecting indexed packages
Full collectionDaily (2am UTC)Runs all six signal collectors across every service
Service discoveryDaily (4am UTC)Scans registries for new AI services to add to the index

When a new CVE is published that affects an indexed package, the hourly scan picks it up and triggers a score recomputation. Health check results are aggregated into 30-day uptime percentages and latency percentiles for the Operational Health signal.

No manual reviews. No paid placements. Trust scores are fully automated. Providers cannot pay for a higher score, request manual review, or influence their ranking. The only way to improve a score is to improve the underlying signals.

How to read a profile

Every service in the Trust Index has a dedicated profile page. Here's what each section tells you and how to use it.

Score ring

The large circular score at the top right of every profile. The ring fills proportionally to the score (0–5) and is colour-coded by status: Trusted green, Caution orange, or Blocked red. The number in the centre is the composite score to two decimal places. Below it you'll see the status label and a timestamp showing when the score was last recalculated.

Signal breakdown

Six horizontal bars showing each signal's individual score (0–5) alongside its weight. This is where you look to understand why a service scored the way it did. A service might have a great overall score but a weak Maintenance Activity signal — meaning it's good today but may not be actively maintained. Bars animate on scroll to draw attention to relative strengths and weaknesses.

Privacy & compliance grid

A six-cell grid showing binary compliance signals: Training Opt-out, Retention Policy, SOC 2, GDPR, Model Card, and HIPAA. Each cell shows a green checkmark (confirmed), red X (not available), or grey "Unknown" (insufficient data). These are not factored into the trust score — they're supplementary context for your own evaluation.

Incident timeline

A chronological log of notable events affecting the service. Each entry is colour-coded: red for incidents, orange for degradation, blue for updates, and green for resolutions. Where possible, we show the score impact of each event (e.g. "−0.34" after a major outage, "+0.12" after a vulnerability patch). This helps you see how a score has changed over time and why.

Score history chart

A 90-day line chart showing the service's composite score over time. The gradient fill below the line is colour-coded by status. Hover over any point to see the exact score and date. Look for sudden drops (incidents), gradual declines (abandonment), or steady climbs (active improvement).

Community metrics

Aggregate usage data pulled from public sources: total downloads or runs, known integrations with agent frameworks, and community ratings from registries. These numbers provide context for the Adoption signal — but they're shown separately so you can evaluate them independently.

Data sources

A list of every source we pulled data from to generate this profile's score. Each entry shows the source name and key stats (e.g. "GitHub · 34.2k ★ · 412 contributors"). This is our transparency layer — you can verify any claim by clicking through to the original source.

Category-normalised scoring. The Adoption signal uses logarithmic normalisation against category peers. A niche MCP tool with 500 downloads isn't penalised against an LLM with 50 million — it's compared against other MCP tools. This prevents category bias in the overall score.

Fabric Monitor is first-party data. Every other source above is public — anyone could build a scoring system from OSV + GitHub + npm. Fabric Monitor is the only signal generated by our own infrastructure. We actively ping service endpoints, measure latency, and detect downtime. No provider can fake this data because they don't control the measurement.

Source reliability

Not all sources are always available. When a source is temporarily unreachable (e.g. GitHub API rate limit), Fabric uses a fallback value rather than zero — this prevents a temporary data collection failure from unfairly tanking a score. The score is flagged as "stale" for that signal until fresh data is collected on the next run.

Claim your profile

Every service in the Trust Index is indexed automatically — publishers don't need to opt in. But claiming your profile unlocks additional capabilities and gives you more control over how your service appears.

What claiming gives you

  • Verified publisher badge — a blue checkmark confirming you own this service, visible on your profile and in directory cards
  • Enhanced profile — add a custom description, logo, documentation links, and pricing information
  • Incident response — get notified when your score changes significantly and respond to incidents directly on your profile
  • Score insights — detailed breakdown of what's helping and hurting your score, with actionable recommendations
  • Priority re-scoring — request an immediate score refresh after fixing a vulnerability or resolving an outage

How to claim

  1. Navigate to your service's profile page on the Trust Index
  2. Click "Claim Provider" at the bottom of the page
  3. Verify ownership via one of: DNS TXT record, GitHub repository admin access, or package registry maintainer status
  4. Once verified, your profile is marked as claimed and you can manage it from the publisher dashboard

Claiming is free. There is no cost to claim your profile. Claiming does not influence your trust score in any way — it only affects the supplementary profile information and gives you access to insights and notifications.

Improving your score

Trust scores are fully automated. There is no manual review process, no paid placement, and no way to influence your score other than improving the underlying signals. Here's what actually moves the needle.

Highest impact (Vulnerability & Safety — ×0.25)

  • Patch known CVEs promptly — critical unpatched vulnerabilities deduct 2.0 points from this signal and floor the score at 1.0
  • Audit your dependency tree — transitive vulnerabilities are counted at 25% of direct rates, capped at −0.25 total
  • Remove unnecessary dependencies to reduce attack surface
  • Add a SECURITY.md with a responsible disclosure process (contributes to Transparency signal)

High impact (Maintenance — ×0.20)

  • Commit regularly — last commit within 7 days scores 5.0, within 30 days scores 4.0, over 365 days scores 1.0
  • Release regularly — monthly release cadence adds +0.3 to the signal
  • Respond to issues — median response time under 24 hours adds +0.5
  • Don't go quiet — fewer than 5 commits in 90 days triggers steep maintenance penalties

Moderate impact (Operational Health — ×0.15)

  • Maintain 99.9%+ uptime — scored from health checks that run every 15 minutes
  • Ensure your endpoint URL is reachable — services with a real endpoint can score up to 5.0; registry proxy only caps at 4.5
  • Keep latency consistent — p99 spikes reduce the score

Moderate impact (Adoption — ×0.15)

  • Grow real usage organically — inflated downloads are detectable and the signal is category-normalised
  • Download growth >20% week-over-week adds +0.5 to the signal
  • Adoption is scored on logarithmic tiers: ≥10M weekly downloads → 5.0, ≥100K → 4.0, ≥1K → 3.0

Moderate impact (Transparency — ×0.15)

  • Open source your code — public source repository is the first checklist item
  • Use a recognised OSI licence
  • Write proper documentation with code examples
  • Publish API documentation
  • For AI models: publish a model card (this is an additional checklist item for LLM, image, speech, and vision categories)

Lower impact (Publisher Trust — ×0.10)

  • Use an organisation account (scores 0.75) rather than a personal account (scores 0.4)
  • Build a track record — account age ≥5 years scores 1.5 (max), ≥2 years scores 1.2
  • Maintain multiple public repositories — ≥20 repos scores 0.75
  • Keep identity consistent across registries (GitHub username matches npm publisher)

You cannot pay for a higher score. Not now, not ever. The only way to improve is to genuinely improve the underlying signals. This is the entire point of Fabric — scores that mean something because they can't be bought.

FAQ

Is the Trust Index free?

Yes. The Trust Index website is completely free to browse. Every score, signal breakdown, incident history, and data source is publicly accessible with no account required.

How often are scores updated?

Health checks (uptime/latency) run every 15 minutes. CVE scanning runs hourly. All six signal collectors run a full sweep daily at 2am UTC. When the daily collection completes, scores are automatically recomputed with the latest data. This means most scores reflect data that is at most 24 hours old, with vulnerability and uptime data being significantly fresher.

Can providers game the score?

It's extremely difficult. Gaming one signal (like inflating download counts) has minimal effect because five other independent signals would need to be simultaneously spoofed. The highest-weighted signal (Vulnerability & Safety at ×0.25) is sourced from the OSV.dev CVE database that providers can't manipulate. Operational Health is measured by Fabric's own monitoring infrastructure — providers don't control the measurement. Additionally, the override system automatically blocks any service with an unpatched critical CVE, regardless of how strong its other signals are.

What if my score is wrong?

Scores are algorithmically generated and may occasionally reflect stale data or edge cases. If you believe your score is inaccurate, you can: (1) claim your profile and request a priority re-score, or (2) use the "Report Issue" button on any profile page. We investigate all reports. Score adjustments only happen through updated data — the system has safety overrides that can cap scores downward (e.g. for critical CVEs), but no mechanism to manually inflate them.

Does being in the index require my consent?

No. The Trust Index scores services using publicly available data. It operates similarly to a search engine — if your service has a public GitHub repository, a package on npm, or a listing on Hugging Face, it may be indexed automatically. You can request delisting by contacting us, though this does not remove the underlying public data.

How are services discovered and added?

Fabric runs automated discovery scans daily against package registries (npm, PyPI) and GitHub, searching for AI-related packages, MCP servers, and agent tools. High-priority services are also manually linked to ensure comprehensive coverage of the most important tools. When a new service is detected, it enters the scoring pipeline and appears in the index once initial data collection is complete — typically within 24–48 hours. Users can also submit services for inclusion via the Trust Index.

What does "Blocked" actually mean?

A Blocked score means Fabric detected significant risk signals — typically active vulnerabilities, malware indicators, or a combination of abandonment and security issues. It does not mean the service is malicious in every case, but it does mean we strongly recommend additional due diligence before using it.
